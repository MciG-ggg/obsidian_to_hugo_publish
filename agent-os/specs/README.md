## 规范文档结构

每个规范目录包含以下标准化文件：

```
[spec-name]/
├── spec.md              # 完整技术规范
├── project-tracker.md   # 项目跟踪和状态
├── planning/            # 规划阶段文档
│   ├── initialization.md  # 初始需求描述
│   ├── requirements.md   # 详细需求分析
│   └── visuals/         # 设计视觉资产
└── development/         # 开发阶段文档
    ├── tasks.md         # 详细任务清单
    ├── progress.md      # 开发进度跟踪
    └── tests.md         # 测试计划和结果
```

## 贡献指南

### 新建规范流程

1. **创建规范目录**
   ```bash
   mkdir agent-os/specs/YYYY-MM-DD-feature-name
   ```

2. **初始化规划文档**
   ```bash
   cp template/ agent-os/specs/YYYY-MM-DD-feature-name/
   ```

3. **填写初始需求**
   编辑 `planning/initialization.md`

4. **启动需求调研**
   使用 spec-research 代理进行需求分析

5. **完成技术规范**
   使用 spec-shaper 代理完成详细规范

### 规范审查标准

- **需求完整性**: 用户需求是否清晰完整
- **技术可行性**: 技术方案是否可行合理
- **集成兼容**: 与现有系统的兼容性
- **风险可控**: 风险识别和缓解措施是否充分
- **计划合理**: 开发计划是否现实可行
